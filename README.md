# human-activity-recogniser
🏃 Human Activity Recognition: From Sensors to Insights📌 OverviewThis project is an end-to-end Machine Learning pipeline that classifies human physical activities based on smartphone sensor data. While most ML projects stop at a Jupyter Notebook, this project includes a fully deployed web dashboard that simulates a production-level inference engine.🎯 The ChallengeThe core difficulty in Human Activity Recognition (HAR) is the "Signal-to-Action" gap. Raw accelerometer and gyroscope data are noisy and high-frequency. This project demonstrates how to bridge that gap using Digital Signal Processing (DSP) and Regularized Logistic Regression.🏗️ System ArchitectureThe pipeline follows a rigorous scientific approach to data transformation:Data Acquisition: Tri-axial $A_x, A_y, A_z$ signals recorded at $50\text{ Hz}$.Pre-processing: Application of a median filter and a $20\text{ Hz}$ 3rd order low-pass Butterworth filter.Feature Engineering: Sliding window segmentation (2.56s) to extract 561 features across:Time Domain: Mean, Standard Deviation, Signal Magnitude Area (SMA), etc.Frequency Domain: Fast Fourier Transform (FFT) components to identify rhythmic gait patterns.Inference: A Multi-class Logistic Regression model ($L2$ Regularized) trained on the UCI HAR Dataset.🚀 Live DemoAccess the Web App here: [Your Streamlit Link]How to use:Step 1: Download the sample_har_data.csv provided in the app.Step 2: Upload the file to the dashboard.Step 3: View the predicted activity distribution and classification summary.🛠️ Technical Deep DiveWhy Logistic Regression ($L2$)?During the research phase (documented in the Jupyter Notebooks), multiple models were tested. We selected Logistic Regression with L2 Regularization (Ridge) because:Multicollinearity: Sensor features (like Mean and SMA) are often highly correlated. $L2$ helps stabilize the coefficients.Efficiency: For edge deployment (like an Android app), Logistic Regression is computationally lightweight compared to Deep Learning, ensuring low battery drain on mobile devices.Class Imbalance & Confusion MatrixSpecial attention was paid to the "Sitting" vs. "Standing" classes. By analyzing the Confusion Matrix, we identified subtle feature overlaps and optimized the decision boundaries to reduce False Negatives.📂 Project StructurePlaintext├── app.py                # Streamlit Web Interface
